## A04 – Discover a vulnerable website

For this activity, I looked at a website that felt relevant to me because I had previously worked in car-rental administration and had seen how important websites are in daily business operations. That experience made me curious about what happens behind the normal front-end of a business website, especially when there is no clearly visible dedicated cybersecurity support constantly reviewing it. Because of that, I decided to examine the site more carefully instead of only judging it from its appearance. To do this, I used Nikto in Kali Linux. Nikto is an open-source web server scanner that checks websites for common security weaknesses such as unsafe files, weak configurations, missing protections, and other exposed details. I used the command “nikto -h https://fusionrentals.com.au”, which tells Nikto to scan the target host over HTTPS and return any important findings. I found this tool useful because it gave me a practical way to inspect the website’s security posture instead of only looking at its visible pages.

When I reviewed the Nikto output, I realised that the website did not look obviously broken from the outside, but the scan still showed several weaknesses in its security setup. One of the first findings was that the “X-Frame-Options header” was not present. This matters because it means the website may be more exposed to clickjacking-style attacks, where a page can potentially be loaded inside another malicious frame to trick users into clicking things they do not intend to click. Another important finding was that the site uses TLS, but the “Strict-Transport-Security header” is not defined. From a security point of view, this means the website is missing an extra layer of protection that would force browsers to stay on HTTPS more strictly. The Nikto output also reported that the “X-Content-Type-Options header” was not set, which suggests the browser may have more freedom to interpret content types in unsafe ways. What made the site feel vulnerable to me was not one massive flaw, but the fact that several basic defensive headers were missing at the same time. That suggests the website is not as strongly hardened as it should be.

The rest of the Nikto output also gave useful information about the site’s internal setup. The scan showed that the server is running LiteSpeed, and it exposed extra headers such as “x-litespeed-cache”, “x-litespeed-cache-control”, and “x-litespeed-tag”. These are not attacks by themselves, but they reveal technical details about the site’s environment, which can help attackers understand what kind of system they are dealing with. Nikto also found a “Drupal Link header”, which suggests the site may be using Drupal or exposing Drupal-related metadata. Another line showed an “alt-svc header” advertising HTTP/3, but Nikto could not test HTTP/3 over QUIC in that scan. I understood these results as signs that the website reveals more information than necessary about its backend setup. Altogether, the scan told me that this website may be vulnerable not because it is fully compromised already, but because it is missing several standard security protections and is exposing useful technical details that could make future targeting easier. What I learned from this activity is that a website can look perfectly normal to ordinary users while still having weaknesses underneath, and a tool like Nikto is useful because it helps reveal those hidden issues quickly and clearly.
