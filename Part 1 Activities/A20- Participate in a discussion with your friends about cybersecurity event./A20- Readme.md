## A20- Participate in a discussion with your friends about cybersecurity event.


While researching for my cybersecurity portfolio, I came across a very interesting 2024 AWS-related case that immediately caught my attention. At first, I thought it was going to be a normal story about a big cloud provider getting hacked, but the more I read, the more unusual and educational it felt. As part of the portfolio instruction, I decided to share it with my friend Joy and discuss it with him, because it seemed like the kind of incident that connects very well to the things we learn in cybersecurity. What made the case so interesting to both of us was that AWS itself was not the main thing being hacked. Instead, attackers were finding publicly exposed .env files on badly configured websites and servers, then stealing the AWS credentials stored inside them. Unit 42 explained that the attackers targeted around 110,000 domains and collected over 90,000 unique environment variables, which made us realise how dangerous careless secret storage can be. 

As Joy and I kept talking about it, we started understanding the attack chain much better. Once the attackers found exposed AWS-related credentials, they could use them for initial access into cloud environments. From there, they carried out discovery and reconnaissance, meaning they checked what those compromised accounts were allowed to do and what resources they could reach. When Joy shared one of the articles with me, what stood out to both of us was that this was not really about some dramatic exploit against AWS itself, but about attackers taking advantage of misconfiguration, exposed secrets, and poor credential handling. That made the case feel much more realistic and useful from a learning point of view, because it showed how a small mistake in setup can lead to a very large incident. 

One of the most interesting moments in our discussion was when Joy asked a really intriguing question: if an attacker is using real AWS credentials and normal cloud services, at what point does the activity stop looking like normal admin work and start looking like an attack? I found that question genuinely thought-provoking because it gets to the heart of cloud security. We both realised that attackers do not always need to behave in an obviously suspicious way at first. If they already have valid keys, they can use real APIs and normal cloud features, which can make malicious behaviour look deceptively legitimate. That is why this case kept pointing us back to ideas like logging, monitoring, anomaly detection, and behaviour analysis. It helped us see why tools such as CloudTrail logs and cloud monitoring systems are so important in modern environments.

Another thing Joy and I discussed was the scale of the incident, which made it feel even more serious. The Cloud Security Alliance summary described the campaign as affecting more than 230 million cloud and SaaS environments through insecurely handled identities and credentials. We both agreed that this made the incident much bigger than just one organisation losing control of one cloud account. It showed how one exposed secret could become the starting point for more scanning, more data theft, and more follow-on attacks. That was one of the main reasons this case felt so exciting to study — it showed how one small weakness can grow into a much larger chain of damage.

By the end of the discussion, we had moved naturally from “what happened” to “what could have prevented it.” From the sources we read, the clearest lessons were to avoid exposing .env files publicly, store secrets more safely, rotate credentials quickly, reduce permissions using least privilege, and monitor cloud activity more carefully. Joy and I both felt that this incident was one of the most interesting examples of modern cloud security because it connected many important ideas in one real case: credential theft, exposed secrets, privilege misuse, lateral movement, extortion, and cloud monitoring. Overall, discussing this case with Joy made the topic much more memorable, because it felt less like reading theory and more like seeing how small security mistakes can turn into a very large real-world incident.
